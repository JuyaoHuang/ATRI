## 七、关键设计原则

1. **chat_history 是 source of truth** — append-only，天然可靠
2. **mem0 天然幂等** — v3 ADD-only，重复写入无害
3. **接口层只抛异常** — 重试由调用方决定
4. **宁可漏删不可误删** — L1 填充词表原则
5. **恢复逻辑 = 正常逻辑** — 不引入专门的恢复代码路径
6. **模块间通过接口依赖** — 不依赖具体实现类
7. **配置与代码分离** — YAML 文件管配置，代码不硬编码

- **`uv run ruff format` 不支持 `.md`**：markdown 格式化是 experimental，只对 `.py` 跑 format。
- **`yaml.safe_load` 空 frontmatter → None**：`or {}` 兜底 + `isinstance(meta, dict)` 拒绝 YAML list/scalar。
- **`monkeypatch._PERSONA_DIR`**：prompt_loader 的 `_PERSONA_DIR` 是模块级属性，测试直接 setattr 即可隔离。
- **`append_system_note` 同步设计**：`ChatHistoryWriter.append_system` 是普通文件 I/O，async 语义无价值；同步让"无轮次副作用"契约更直观（async 天然暗示状态推进）。
- **module-level monkeypatch `"src.memory.manager.l3_collapse"`**：manager.py `from src.memory.compressor import l3_collapse` 把函数名绑到模块属性，monkeypatch 按字符串路径可替换。
- **`recent_messages is seed`（identity check）**：比 `== seed` 更严格——捕捉"意外替换列表对象"bug，原地 mutation 仍识别为破坏不变式。
- **async 生成器 mock 模式**：`MagicMock(side_effect=lambda *a, **kw: _astream(chunks))`。`side_effect=lambda` 保证每次调用新建 gen（避免 exhausted reuse）；`AsyncMock(return_value=...)` 只适用于 `await mock()`，`async for` 需同步返回 AsyncIterator。
- **`async def` + `yield` 即 async generator**：`agent.chat(x)` 直接返回 gen（**非**协程）；`async for chunk in agent.chat(x):` 即可；`return` 在 async gen 中触发 `StopAsyncIteration`。
- **Error path 的 partial stream 语义**：`try/except LLMError` 包住整个 `async for`，已 yield chunks 保留给调用方；`return` 跳过 `on_round_complete`，不入记忆；`yield error_text` 在 `except` 块里触发。
- **非 LLMError 透传**：`except LLMError` 精确捕获基类及子类，`ValueError` 等无关异常自然向外抛。
- **LLMError 三子类统一捕获**：`LLMConnectionError` / `LLMRateLimitError` / `LLMAPIError` 都继承 `LLMError`，一句覆盖；parametrize 测试验证。
- **PRD 的 `LLMFactory.create_from_role` 过时**（CLAUDE.md Step 9 触发）：PRD US-AGT-005 写 `LLMFactory.create_from_role(...)`，实际 API 是 `src.llm.factory` 模块级函数 `create_from_role`（`LLMFactory` 类只有 `register/create/available` 三方法）。**教训**：PRD 是意图指导，API 名必须以代码为准——写代码前先 grep。写完模块立即 mypy 也能抓到这类 attribute 错误。
- **ServiceContext cache key 必须是 tuple**：`(character_id, user_id)` 而非字符串拼接——Python dict 原生支持 tuple key，语义清晰；字符串拼接需自选分隔符，易与合法输入冲突。
- **`_safe_build_long_term` 捕获 `Exception`**：`local_deploy` 模式下 `Memory.from_config` 可能 prompt 安装 ollama 库（stdin 读一行），无 tty 场景会 `EOFError`；宽 `except Exception` 覆盖此类非预期失败。
- **ChatAgent 的 `system=` 不要重复前置**：`openai_compatible.py` 仅在 `system is not None` 时前置 system role；`build_llm_context` 已把 system 放位置 [1]，ChatAgent 调 `chat_completion_stream(messages)` **不传** `system=`，避免重复。
- **DeepSeek-V3.2 / SiliconFlow streaming 偶发截断**（真实事件）：10 轮真对话 + 2 probe 场景下观察到 Round 10 和 Probe 1 都只返回 `（`（1 字符）就 stream 正常 close，**无异常**。根因：服务端 `finish_reason=length/content_filter` 提前结束，但 provider 未读此字段。应对：生产层只加 WARNING log 观察（c39833b），测试层加 retry + memory fallback（1221625）；**不改 Phase 2 provider**（§七禁止项）。
- **pytest AssertionError 不是代码 bug**：断言失败是"告诉 pytest 的那个表达式求值为 False"，与"链路是否通"、"memory 是否对"无关——`assert "Alice" in "（"` → False → raise AssertionError → pytest FAIL。加 retry 扩大有效样本；加 fallback 把"验证 LLM 能用记忆"降级到"验证记忆层本身持有事实"，后者才是我们代码的职责边界。
- **OLV 同款 gap**：`refer-projects/Open-LLM-VTuber` 的 `stateless_llm/openai_compatible_llm.py` 也无 `finish_reason` 检查，遇到同款截断也无保护；OLV 的 empty-reply 防御（`_add_message` 里 `if not text_content and role == "assistant": return`）只覆盖**完全空回复**，不覆盖"1 字符"截断。参考 OLV 证实"finish_reason 检查非业界通行做法"，用户选 tests 层 retry + 生产层 WARN log 方案工程合理。
- **`git add src/main.py` 可能带入已 staged 的其他文件**：US-AGT-006 commit `c1ac026` 意外带入 `atri/执行准则.md` 的 `## 代码审查` 章节——内容对齐 Phase4_执行规格.md §六，语义正向，用户选保留现状；后续 US-007 commit `1221625` 也意外带入 `tests/agent/chat-history.txt`（旧 live 失败 log），用户同样选保留。经验：`git add <path>` **只追加**指定路径，不清理 index 里已有 staged 改动，commit 前应显式 `git status --short` 检查；或使用 `git commit -- <path>` 限定 commit 范围。
