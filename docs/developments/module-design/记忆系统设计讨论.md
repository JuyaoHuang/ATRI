# Emotion-Robot 记忆系统设计文档

> 本文档为记忆系统的设计蓝本，基于 Open-LLM-VTuber / Neuro / airi 三个参考项目 + mem0 框架 + Claude Code 上下文压缩策略的综合分析，经多轮讨论确定。后续实现应以本文档为准。

---

## 1. 需求锚点

### 1.1 核心目标

构建一个**记忆可持续化**的情感聊天机器人，使其能够：

- 在同一会话内维持连贯的上下文（短期记忆）
- 跨会话记住用户的偏好、事实、情感变化（长期记忆）
- 在对话轮次增长时自动压缩历史，控制 LLM 调用的 token 消耗

### 1.2 响应管道中的记忆位置

```
用户输入(文字/ASR) → L1 Snip清洗 → 短期记忆管理 → mem0长期记忆检索
                                                          ↓
                                              构建 LLM Context
                                                          ↓
                                                    LLM 调用
                                                          ↓
                                              LLM 实时输出文本
                                                    ↓         ↓
                                                  TTS      翻译模块
                                                    ↓
                                              写入 chat_history
                                              更新 short_term_memory
```

### 1.3 设计约束

- 不使用 MCP、Plugin 等外部工具，专注 RAG 系统
- 支持双模式部署：mem0 SDK 云端调用 + 本地开源部署
- TTS/翻译模块消费 LLM 实时输出流，不读取存储文件
- 前端负责读取 chat_history 渲染历史消息，后端负责记忆管理

## 2. 参考来源分析

### 2.1 各项目对记忆系统的贡献

| 来源 | 贡献点 | 采纳情况 |
|------|--------|----------|
| **Neuro** | 反思式记忆：每 20 条消息生成 3 个 Q&A 写入 ChromaDB | 借鉴"定期批量提取"的思路，但用 mem0 替代 ChromaDB，用自由事实抽取替代固定 Q&A |
| **airi** | 双栈存储：浏览器端 DuckDB + 服务端 pgvector | 借鉴"分层存储"理念，但简化为文件(短期) + mem0 向量库(长期) |
| **Open-LLM-VTuber** | 工厂模式的模块化设计、chat_history JSON 格式 | 采纳 JSON 存储格式（调整为一个 session 一个文件），参考模块化架构 |
| **mem0** | v3 ADD-only 事实抽取、三路融合检索、多租户隔离 | 作为长期记忆的核心基座，提供 `add()` 写入 + `search()` 检索 |
| **Claude Code** | 4 级渐进式上下文压缩（Snip/Micro/Collapse/Auto Compact） | 采纳 3 层压缩策略（L1+L3+L4），轮次驱动替代 token 驱动 |

### 2.2 关键设计决策对比

| 设计点 | Neuro 做法 | 我们的做法 | 为什么 |
|--------|-----------|-----------|--------|
| 记忆写入时机 | 每 20 条消息反思 | 每 26 轮触发 L3 时批量写入 mem0 | 频率接近，但我们的"轮"定义更严格（一问一答对） |
| 记忆抽取方式 | 固定问"3 个最显著问题+答案" | mem0 v3 自由事实抽取 | 自由抽取覆盖面更广，不受固定模板限制 |
| 短期记忆结构 | 最近 5 条 history + 全部 twitch 消息 | 分层 block 结构 + 最近 6 轮 raw | 支持长对话，不会因轮次增长导致 context 爆炸 |
| 向量库选择 | ChromaDB PersistentClient | mem0 可插拔（Qdrant/Chroma/pgvector 等 30 家） | 灵活性更高，可按需切换 |
| 多租户 | 单集合，无隔离 | mem0 原生 user_id/agent_id/run_id 三维隔离 | 天然支持多用户 × 多角色 |

## 3. 短期记忆设计

### 3.1 概述

短期记忆负责管理**当前会话内**发送给 LLM 的上下文内容。核心挑战：随着对话轮次增长，原始消息的 token 量会线性膨胀，必须通过压缩控制发送体积。

采用仿 Claude Code 的 **3 层渐进式压缩策略**，以对话轮次（而非 token 预算）作为触发条件。

### 3.2 轮次定义

**1 轮 = 一个完整的 user→ai 交互对。**

计数规则：
- 用户发送一条消息 + AI 返回一条有效回复 = 1 轮
- AI 回复为空或为错误信息（如 `Error calling the chat endpoint`）→ 不计入轮数
- 用户连续发送多条消息（ASR 碎片）→ 由 L1 合并为一条，等 AI 回复后才算 1 轮
- system 消息（中断标记等）→ 不计入轮数，但保留在 chat_history 中

```python
def count_rounds(messages):
    """计算有效轮数"""
    return len([
        m for m in messages
        if m["role"] == "ai"
        and m["content"]                          # 非空
        and not m["content"].startswith("Error")   # 非错误
    ])
```

### 3.3 三层压缩策略

#### 总览

| 层级 | 名称 | 触发条件 | 处理内容 | 动用 LLM | 来源 |
|------|------|----------|----------|----------|------|
| **L1** | **Snip** | 每轮自动 | 纯规则清洗：去填充词、去重、截断超长 | 否 | Claude Code L1 |
| **L3** | **Collapse** | 每 26 轮 | 压缩最早 20 轮 → 生成 block | 是 | Claude Code L3 |
| **L4** | **Super-Compact** | 每累积 4 个 block | 合并 4 个 block → 生成 meta_block | 是 | Claude Code L4 |

> 跳过 Claude Code 的 L2（Micro 压缩），因为 L2 的核心价值是"不破坏 prompt cache key"，emotion-robot 没有 prompt cache 机制，L2 在此场景无意义。

#### L1 Snip — 每轮自动，纯规则

**作用范围：仅处理用户消息，不修改 AI 回复。**

AI 的回复由 system prompt 控制（包括口头禅、角色个性），不应被清洗规则干扰。

| 规则 | 操作 | 示例 |
|------|------|------|
| 填充词去除 | 正则匹配可配置词表，删除无意义词 | "嗯那个我想问一下就是说天气" → "我想问一下天气" |
| 连续重复去重 | 相邻 2+ 条相似度 > 阈值的用户消息，保留最后一条 | 用户说了 3 次"你好" → 保留 1 次 |
| 超长截断 | 单条消息超过阈值时截断，保留前 N tokens + "[已截断]" | 用户粘贴大段文章 → 保留前 800 tokens |
| 系统消息剥离 | 去掉非对话内容（心跳、Live2D 状态同步） | `[heartbeat] alive` → 删除 |

**对所有用户输入统一执行**（不区分 ASR/文字输入），延时 < 1ms，可忽略。

配置项（`memory_config.yaml`）：

```yaml
snip:
  filler_words: ["嗯", "啊", "呃", "额", "那个", "就是说", "对对对"]
  similarity_threshold: 0.95
  max_single_message_tokens: 800
```

#### L3 Collapse — 每 26 轮触发，动用 LLM

**触发条件：** 当前有效轮数达到 26 的倍数时触发。

**处理流程：**

```
当前 26 轮消息（经 L1 清洗后）
  │
  ├─ 最早 20 轮 ──→ 压缩器 LLM ──→ block_N（结构化摘要）
  │                                    ↓
  │                              存入 short_term_memory.json 的 active_blocks
  │
  ├─ 最早 20 轮 ──→ mem0.add(raw_messages) ──→ mem0 向量库（长期事实）
  │
  └─ 最近 6 轮 ──→ 保留在 recent_messages 中
```

**压缩粒度 — 事件级，保留 4 类信息（按优先级）：**

| 优先级 | 保留内容 | 示例 |
|--------|----------|------|
| P0 | 用户事实 & 偏好 | "用户叫小明，养了一只橘猫叫团子" |
| P1 | 情感状态变化 | "用户从开心变得沮丧，因为加班" |
| P2 | 未完成话题 & 承诺 | "用户说明天要面试，让我提醒" |
| P3 | 话题转折点 | "从聊天气转到聊工作压力" |

**丢弃：** 寒暄、AI 冗长解释（只保留结论）、已被纠正的错误信息、重复讨论的同一话题。

**压缩 prompt 模板（`prompts/l3_collapse.txt`）：**

```
你是一个对话记忆压缩器。请将以下 {N} 轮对话压缩为一段结构化摘要。

要求保留：
1. 用户表达的情感状态变化（开心→沮丧→平静）
2. 用户透露的个人偏好和事实（喜欢猫、住在北京、讨厌加班）
3. 对话中达成的共识或承诺（"明天提醒我"、"下次聊这个话题"）
4. 关键话题转折点

要求丢弃：
1. 寒暄和重复的问候
2. AI 的冗长解释（只保留结论）
3. 已被纠正的错误信息

输出格式：
## 对话摘要 (轮次 {start}-{end})
- 情感轨迹: ...
- 关键事实: ...
- 未完成话题: ...
- 用户偏好: ...
```

**生成的 block 结构：**

```json
{
  "block_id": "block_003",
  "level": 0,
  "covers_rounds": [41, 60],
  "created_at": "2026-04-17T18:00:00Z",
  "summary": "## 对话摘要 (轮次 41-60)\n- 情感轨迹: 平静→兴奋→平静\n- 关键事实: 用户提到下周要出差去上海\n- 未完成话题: 用户想了解上海美食推荐\n- 用户偏好: 偏好清淡口味",
  "token_count": 350
}
```

**压缩比：** 20 轮原始对话（约 2000-4000 tokens）→ 1 个 block（约 300-500 tokens），约 6:1。

#### L4 Super-Compact — 每累积 4 个 block 触发，动用 LLM

**触发条件：** `active_blocks` 中累积 4 个 level-0 block 时触发。

```
block_1 (轮次 1-20)  ─┐
block_2 (轮次 21-40) ─┤→ LLM 合并 → meta_block_1 (轮次 1-80)
block_3 (轮次 41-60) ─┤
block_4 (轮次 61-80) ─┘
```

**压缩粒度 — 模式级（与 L3 的事件级不同）：**

| 优先级 | 保留内容 | 与 L3 的区别 |
|--------|----------|-------------|
| P0 | 稳定的用户画像 | L3 保留单次事实，L4 提炼为持久特征 |
| P1 | 情感变化趋势 | L3 保留"今天沮丧"，L4 提炼"最近一周情绪低落" |
| P2 | 反复出现的话题 | L3 保留单次话题，L4 识别"用户经常聊工作压力" |
| P3 | 关系发展阶段 | L3 没有这个维度，L4 才能看到跨时段的关系变化 |

**核心逻辑：L3 记录"发生了什么"（事件级），L4 提炼"这意味着什么"（模式级）。**

**合并 prompt 模板（`prompts/l4_super_compact.txt`）：**

```
你是一个长期记忆整合器。以下是 4 段对话摘要，覆盖 {total_rounds} 轮对话。
请整合为一段更高层的记忆，重点提取：
1. 跨时段的情感变化趋势（而非单次情绪）
2. 反复出现的话题和兴趣
3. 用户性格特征的稳定模式
4. 关系发展的阶段性变化
```

**压缩比：** 4 个 block（约 1200-2000 tokens）→ 1 个 meta_block（约 400-600 tokens），约 3:1。

**触发频率：** 每 104 轮才触发一次（26 × 4），非常低频。

### 3.4 压缩器模型配置

L3/L4 的压缩模型通过配置文件指定，用户可自主选择：

```yaml
# memory_config.yaml
compressor:
  l3_model: "claude-haiku-4-5"     # L3 频率高，可用轻量模型
  l4_model: "claude-sonnet-4-6"    # L4 频率低，可用更强模型
  # 或统一配置：
  # model: "gpt-4o-mini"
  l3_prompt_template: "prompts/l3_collapse.txt"
  l4_prompt_template: "prompts/l4_super_compact.txt"
```

### 3.5 发送 payload 构成

每次 LLM 调用时，按以下顺序拼接 context：

```
[1] System Prompt（角色设定、口头禅等）
[2] mem0 长期记忆检索结果（mem0.search 返回的相关事实）
[3] meta_blocks（L4 产物，最旧的压缩层）
[4] active_blocks（L3 产物）
[5] 最近 6 轮 raw 消息（经 L1 清洗）
[6] 本轮用户输入
```

**示例（第 130 轮）：**

```
meta_block_1 (轮次 1-80)     ← L4 产物，~500 tokens
block_5 (轮次 81-100)        ← L3 产物，~350 tokens
block_6 (轮次 101-120)       ← L3 产物，~350 tokens
最近 6 轮 raw (轮次 125-130) ← 经 L1 清洗，~1200 tokens
本轮用户输入                  ← ~200 tokens
─────────────────────────────
总计约 2600 tokens（对比发送 130 轮 raw ≈ 26000 tokens）
```

## 4. 长期记忆设计

### 4.1 概述

长期记忆负责**跨会话**持久化用户的事实、偏好、情感模式等信息。采用 mem0 作为核心基座，仅使用其 `add()` 写入和 `search()` 检索能力，不使用 mem0 内置的 messages 表。

### 4.2 mem0 集成方式

#### 双模式部署

| 模式 | 实现类 | 适用场景 | 配置 |
|------|--------|----------|------|
| **SDK 云端** | `MemoryClient(api_key=...)` | 快速验证、无需本地基础设施 | 仅需 API key |
| **本地开源** | `Memory(config=...)` | 生产部署、数据隐私、可定制 | 需配置向量库 + LLM + Embedding |

两种模式共享相同的 API 接口（`add` / `search` / `get_all` / `delete` / `history`），业务层通过一个 `MODE` 配置项切换：

```yaml
# memory_config.yaml
mem0:
  mode: "local"  # "cloud" | "local"
  cloud:
    api_key: "${MEM0_API_KEY}"
  local:
    vector_store:
      provider: "qdrant"       # 可选：qdrant / chroma / pgvector / faiss 等
      config:
        host: "localhost"
        port: 6333
    llm:
      provider: "openai"       # 用于事实抽取的 LLM
      config:
        model: "gpt-4o-mini"
    embedder:
      provider: "openai"
      config:
        model: "text-embedding-3-small"
```

#### 多租户映射

mem0 原生支持 `user_id` / `agent_id` / `run_id` 三维隔离，映射到 emotion-robot：

| mem0 维度 | emotion-robot 含义 | 示例 |
|-----------|-------------------|------|
| `user_id` | 用户标识 | `"alen"` |
| `agent_id` | 角色标识 | `"katou"` |
| `run_id` | 会话标识 | `"2026-04-17_a3f8"` |

调用示例：

```python
# 写入 — L3 触发时
mem0.add(
    messages=raw_20_rounds,       # 原始 20 轮消息（不是 block）
    user_id="alen",
    agent_id="katou",
    run_id="2026-04-17_a3f8"
)

# 检索 — 每轮 LLM 调用前
results = mem0.search(
    query=current_user_input,
    user_id="alen",
    agent_id="katou",
    limit=5
)
```

### 4.3 写入时机

| 时机 | 触发条件 | 写入内容 |
|------|----------|----------|
| **L3 压缩时** | 每 26 轮 | 最早 20 轮的原始消息 |
| **会话关闭时** | 用户关闭窗口 / 超时断开 | 剩余未被 L3 处理的所有消息（即使不满 20 轮） |

**关键决策：喂给 mem0 的是原始消息，不是 block。**

理由：
- mem0 v3 的事实抽取算法是为原始对话设计的
- 喂 block（二手摘要）会导致 mem0 抽取"摘要的摘要"，事实失真
- mem0 和 block 系统各自独立抽取，互不干扰

会话关闭时的处理：

```python
def on_session_close(session):
    remaining = session.get_uncompressed_messages()
    if remaining:
        mem0.add(
            messages=remaining,
            user_id=session.user_id,
            agent_id=session.agent_id,
            run_id=session.session_id
        )
    # chat_history.json 已在每轮实时写入，无需额外操作
```

### 4.4 检索时机与注入

每轮 LLM 调用前，用当前用户输入做一次语义检索：

```python
def build_llm_context(user_input, short_term_memory, session):
    # 1. 检索长期记忆
    long_term = mem0.search(
        query=user_input,
        user_id=session.user_id,
        agent_id=session.agent_id,
        limit=5
    )

    # 2. 拼接 context
    messages = []
    messages.append({"role": "system", "content": system_prompt})

    if long_term:
        memory_text = "\n".join([m["memory"] for m in long_term])
        messages.append({"role": "system", "content": f"关于这位用户，你记得：\n{memory_text}"})

    for mb in short_term_memory["meta_blocks"]:
        messages.append({"role": "system", "content": mb["summary"]})

    for ab in short_term_memory["active_blocks"]:
        messages.append({"role": "system", "content": ab["summary"]})

    for msg in short_term_memory["recent_messages"]:
        messages.append({"role": msg["role"], "content": msg["content"]})

    messages.append({"role": "user", "content": user_input})
    return messages
```

### 4.5 不使用 mem0 messages 表

mem0 内部的 `messages` 表（`SQLiteManager.save_messages`）是一个固定保留最近 10 条原始消息的缓存，硬编码淘汰策略，不可配置。

**决策：忽略此表。** 理由：
- 我们的短期记忆系统（分层 block + recent_messages）已完整覆盖此需求
- messages 表不参与 `mem0.search()` 的向量检索，对长期记忆无影响
- 多一个存储点只增加一致性维护复杂度

实现方式：调用 `mem0.add()` 时不触发 `save_messages()`（`save_messages` 是独立调用，不被 `add()` 自动触发）。

## 5. 存储布局

### 5.1 目录结构

```
data/
  characters/
    katou/
      sessions/
        2026-04-17_a3f8.json       ← chat_history（归档）
        2026-04-18_b7c2.json
      short_term_memory.json       ← 当前活跃会话的短期记忆
    shizuku/
      sessions/
        ...
      short_term_memory.json
  config/
    memory_config.yaml             ← 压缩配置、mem0 配置、L1 规则
    prompts/
      l3_collapse.txt              ← L3 压缩 prompt 模板
      l4_super_compact.txt         ← L4 压缩 prompt 模板
```

### 5.2 chat_history.json — 对话历史（归档文件）

**职责：** 存储所有原始对话消息的完整记录。前端读取此文件渲染历史消息。

**规则：**
- 一个 session 一个文件，文件名 = `{session_id}.json`
- 只追加写入，不修改已有内容
- 不参与 LLM 调用

**字段定义：**

```json
[
  {
    "role": "metadata",
    "timestamp": "2026-04-17T18:00:00Z",
    "session_id": "2026-04-17_a3f8",
    "character": "katou"
  },
  {
    "role": "human",
    "timestamp": "2026-04-17T18:00:05Z",
    "content": "今天天气真好",
    "raw_input": "嗯那个今天天气真好啊",
    "name": "Alen"
  },
  {
    "role": "ai",
    "timestamp": "2026-04-17T18:00:08Z",
    "content": "我是高性能的！确实适合出去走走呢",
    "name": "Katou",
    "avatar": "katou.png"
  },
  {
    "role": "system",
    "timestamp": "2026-04-17T18:01:00Z",
    "content": "[Interrupted by user]"
  }
]
```

| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| `role` | string | 是 | `"metadata"` / `"human"` / `"ai"` / `"system"` |
| `timestamp` | ISO 8601 | 是 | 消息时间戳 |

### 5.4 会话生命周期

```
[新建会话]
  → 创建 chat_history.json（metadata 行）
  → 创建 short_term_memory.json（空结构）
  → mem0.search() 可用（跨会话的长期记忆）

[对话进行中]
  → 每轮：L1 清洗 → 写入 chat_history → 更新 short_term_memory.recent_messages
  → 每 26 轮：L3 触发 → 生成 block + mem0.add(raw)
  → 每 4 个 block：L4 触发 → 生成 meta_block

[会话关闭]
  → 剩余未压缩消息 → mem0.add()（即使不满 20 轮）
  → short_term_memory.json 保留（下次恢复用）
  → chat_history.json 保留（前端渲染用）

[恢复会话]
  → 前端：读取 chat_history.json 渲染历史消息到窗口
  → 后端：读取 short_term_memory.json 恢复 block 结构 + recent_messages
  → 从上次的 total_rounds 继续计数

[新会话（同角色）]
  → 新建 chat_history.json（新 session_id）
  → 新建 short_term_memory.json（空结构）
  → mem0 长期记忆仍然可用（user_id + agent_id 不变）
```

### 6.1 数据流全景

> **修订注记（2026-04-19，对齐 Phase 3 实现 + Phase 4 ChatAgent 规划）**
>
> 相比早期蓝图，有两处关键变化：
>
> 1. **L1 Snip 从"独立入口节点"内化到 `MemoryManager.on_round_complete` 内部**
>    （manager.py L552-553）。职责归属更清晰：L1 属于记忆层，不暴露为公共 API。
> 2. **引入 `ChatAgent` 组合层**（Phase 4，`src/agent/chat_agent.py`）作为调用
>    LLM 与 MemoryManager 的统一入口。
>
> 数据一致性不变：chat_history / recent_messages / mem0 索引全部基于 L1 清洗版；
> 当轮 `user_input` 直达 LLM 位置 [6] 时保留 raw（当轮鲜活语境）。

```
用户输入（文字 / ASR，raw text）
  │
  ▼
ChatAgent（Phase 4 组合层）
  │  持有: persona + LLMInterface(role=chat) + MemoryManager
  │
  ├── (1) await mgr.build_llm_context(raw user_input, persona)
  │        │
  │        ▼ 内部组装 payload（严格按 §3.5 顺序）：
  │        ┌────────────────────────────────────────────────────┐
  │        │ [1] system                : persona                │
  │        │ [2] 长期事实              : search_long_term(raw)  │
  │        │     └→ mem0.search(query=raw, user_id, agent_id)   │
  │        │        → 包装 "关于这位用户，你记得：\n- ..."       │
  │        │ [3] meta_blocks   (cleaned; L4 产物)                │
  │        │ [4] active_blocks (cleaned; L3 产物)                │
  │        │ [5] recent_messages (cleaned; §3.2 有效轮)          │
  │        │ [6] 本轮 user_input (raw, 原样透传)                 │
  │        └────────────────────────────────────────────────────┘
  │
  ├── (2) async for chunk in llm.chat_completion_stream(messages):
  │        yield chunk   (LLM 原生语言文本，如 zh)
  │            │
  │            ▼ 下游消费者（均为独立 plugin 式模块，启停 + 消费源由配置决定）：
  │            ├→ 前端（默认消费 LLM 原文；可配置改消费翻译后的文本）
  │            ├→ 翻译模块（可选启用；形态:output text -> translate -> translated，
  │            │             启用后其下游消费者 = 前端 / TTS / 两者，由配置决定）
  │            └→ TTS 模块（可选启用；消费源 = LLM 原文 / 翻译后文本，由配置决定）
  │
  ▼ stream 结束（全部 chunks 合并为 reply）
  │
  └── (3) await mgr.on_round_complete(
             user_msg={role:"human", content: raw, raw_input: ASR 原文?},
             ai_msg  ={role:"ai",    content: reply},
         )   ← ChatAgent 自动调用（S1b 决策）
          │
          ▼ MemoryManager.on_round_complete 内部：
          │
          ├── L1 Snip（仅作用于 user_msg；AI 回复不动）
          │   → cleaned_user
          │
          ├── chat_history.append_human(cleaned.content, raw_input=raw_input?)
          ├── chat_history.append_ai(reply)
          │     （错误轮次 S4 改走 append_system_note，不走 append_ai）
          │
          ├── recent_messages += [{human: cleaned}, {ai: reply}]
          │
          ├── total_rounds++（仅当 _is_valid_round：content 非空 &
          │                    不以 "Error" 开头；§3.2）
          │
          ├── 检查 total_rounds % 26 == 0?
          │     └─ 是 → L3 Collapse
          │              ├→ 最早 20 轮 cleaned → 压缩 LLM → block → active_blocks
          │              └→ 同一窗口 → long_term.add()
          │                 ↓ mem0 内部调 LLM 抽事实 → 事实句 embedding 入索引
          │
          └── 检查 active_blocks ≥ 4?
                └─ 是 → L4 Super-Compact
                         └→ 4 blocks → 压缩 LLM → meta_block → meta_blocks
```

**关键事实（呼应实现）**：

1. **raw text 的去处**：当轮 `user_input` 走两条"raw 路径"——直达 LLM 位置 [6]、作为 `search_long_term` 的 query；其余所有落盘路径（chat_history.content / recent_messages / mem0.add 的 window）都经 `on_round_complete` 内部 L1 得到 cleaned。

2. **mem0 索引的本质**（详见 `docs/projects-docs/mem0_架构文档.md` §3.2/§3.3）：`mem0.add` 内部会调一次 LLM 抽取事实句，索引的是**事实句 embedding**（如"用户最爱喝珍珠奶茶"），而非 add 入参 messages 的 embedding。这使 search query 的 raw vs cleaned 差异在索引维度被吸收。

3. **翻译模块 与 TTS**：两者均为**独立的 plugin 式模块**，共同作为 LLM stream 的旁路消费者——**不入 chat_history**、**不影响 on_round_complete**。
   - **翻译模块**：可选启用（配置开关），形态 `output text → translate module → translated text`。启用后其下游消费者由配置决定（前端 / TTS / 两者），**本身不决定谁消费**。
   - **TTS 模块**：同样独立可选，消费源由配置决定——LLM 原文（默认）或翻译模块输出。
   - **前端**：默认消费 LLM 原文；可配置改消费翻译后文本。
   - chat_history 始终记录 LLM 原生语言，记忆 / 压缩的语言一致性不受翻译 / TTS 启停影响。

4. **错误路径（S4）**：LLM 抛 `LLMError` 时 ChatAgent yield 错误文本给调用方，并调 `mgr.append_system_note("[LLM call failed: ...]")` 写入 chat_history 的 `role=system` 行——**不计轮、不触发 L3、不入 recent_messages**。

### 8.1 Item #1 向量库选型

**决策：** 本地 **Qdrant**（双模式：嵌入式默认 + Server 可选）；云端 **mem0 SaaS SDK**。多角色隔离交给 mem0 原生 `agent_id` filter，无需应用层分 collection。

**Qdrant 两种部署模式对比：**

| 维度 | 嵌入式（文件模式） | Server 模式（Docker） |
|---|---|---|
| 本质 | Python 库跑在主进程内，数据存本地目录 | 独立服务进程，HTTP/gRPC 连接 |
| 类比 | SQLite | PostgreSQL |
| 启动 | `QdrantClient(path="./qdrant_data")` | `docker run qdrant/qdrant` + `QdrantClient(host=...)` |
| 安装成本 | `pip install qdrant-client` | 需 Docker |
| 并发写入 | ❌ 不支持（进程锁） | ✅ 支持 |
| 远程访问 | ❌ | ✅ |
| 上云迁移 | 数据需导入导出 | ✅ 改 host 地址即可 |
| 代码差异 | 只改 `QdrantClient` 构造参数，业务逻辑一致 | 同左 |

**最终选型理由：**
- MVP/自用默认嵌入式，零配置，开箱即用
- 上云或多用户并发时切 Server 模式，代码层零修改（mem0 的 `vector_store.provider` 抽象）
- `sdk` 模式下此项不生效（mem0 SaaS 托管向量库）

### 8.2 Item #2 Embedding / LLM / 部署结构

**核心认知：** `mode` 表示"mem0 的使用形态"，与代码部署位置正交（本地也能跑 `sdk`，VPS 也能跑 `local_deploy`）。

**最终配置结构（取代 §4.2 和 §7 中的原版示例）：**

```yaml
mem0:
  mode: "local_deploy"    # "sdk" | "local_deploy"

  # ─── 模式 A：mem0 SaaS（托管）───
  sdk:
    api_key: "${MEM0_API_KEY}"
    # Embedding / LLM / 向量库全部 mem0 托管，用户透明，无需配置

  # ─── 模式 B：本地/自托管 ───
  local_deploy:
    vector_store:
      provider: "qdrant"
      config:
        path: "./data/qdrant"         # 嵌入式默认
        # host: "localhost"            # Server 模式切换
        # port: 6333

    embedder:
      backend: "ollama"                # "ollama" | "api"，默认 ollama
      ollama:
        model: "bge-m3"                # 1024 维，多语言 SOTA
        base_url: "http://localhost:11434"
      api:
        provider: "openai"             # OpenAI / 兼容 API（智谱/阿里/DeepSeek）
        model: "text-embedding-3-small"
        api_key: "${OPENAI_API_KEY}"
        base_url: "https://api.openai.com/v1"

    llm:
      reuse_main_llm: false            # 默认独立配置（冷/热模型分离）
      backend: "ollama"                # "ollama" | "api"，默认 ollama
      ollama:
        model: "qwen2.5:7b"
        base_url: "http://localhost:11434"
      api:
        provider: "openai"
        model: "gpt-4o-mini"
        api_key: "${OPENAI_API_KEY}"
        base_url: "https://api.openai.com/v1"
```

**关键设计点：**

1. **Embedder 三种后端**：`ollama`（本地统一，推荐）/ `api`（OpenAI 兼容，云端/无 GPU 场景）/ 可回退 HuggingFace 直连。mem0 原生支持以上全部。
2. **Embedding 维度对齐**：bge-m3=1024，OpenAI small=1536。换模型需重建向量库（mem0 要求 vector_store 与 embedder 维度一致）。
3. **本地 GPU 上 bge-m3 的性能**：RTX 3060/4060+ 单条 5-15ms，20 轮批压缩仅 200-500ms，完全无压力。
4. **LLM `reuse_main_llm` 默认 false**：默认独立配置，主聊天可用 Claude/GPT，事实抽取用便宜的 qwen2.5/gpt-4o-mini；需要统一时改 `true` 即可复用主 LLM 客户端。
5. **mem0 原生 Embedder Provider**：`openai` / `ollama` / `huggingface` / `vertexai` / `gemini` / `aws_bedrock` / `together` / `langchain`。
6. **Ollama 对比 HuggingFace 直连**：Ollama 作为独立守护进程，embedder+LLM 统一管理、显存独立；HuggingFace 直连依赖 `sentence-transformers` + `torch`，与主进程共显存。

**项目内 LLM 调用的 4 个出口**（设计时需注意）：
1. 主聊天（user→AI）
2. L3 压缩（每 26 轮）
3. L4 super-compact（每 4 个 block）
4. mem0 事实抽取（每次 `mem0.add()`）

`reuse_main_llm: true` 时 #4 复用 #1；默认 false 时 #4 独立；#2/#3 由 `short_term.compressor` 单独指定。

### 8.3 Item #3 `mem0.search()` 参数

**决策（MVP 初值）：** `limit=5 / threshold=0.3 / rerank=false`（均衡版）

**参数选型推理：**

| 参数 | 值 | 推理 |
|---|---|---|
| `limit` | 5 | 每条记忆平均 ~40 tokens，共 ~200 tokens 注入 context；不稀释 system prompt 的角色设定 |
| `threshold` | 0.3 | 情感对话 query 短（20-50 tokens），语义易泛化；偏低阈值防止漏召弱关联的情感记忆（"今天累" ↔ "上次加班"） |
| `rerank` | false | MVP 阶段减少延时与 API 调用成本；上线后若召回精度不足再开启 |

**配置落位：**

```yaml
mem0:
  # ... (mode / sdk / local_deploy 配置见 §8.2)
  search:
    limit: 5
    threshold: 0.3
    rerank: false
```

**兼容性说明：** `sdk` 和 `local_deploy` 两种模式下 `limit` / `threshold` 参数语义一致；`rerank` 在 SaaS 端可能由 mem0 内部算法替代（非用户可控），`local_deploy` 下需额外配 reranker provider 才能启用。

**调优触发条件**（留给上线后实测）：
- 用户反馈"AI 记不住我说过的话" → 降低 threshold 或提高 limit
- 注入的长期记忆与当前话题无关 → 提高 threshold 或启用 rerank
- 响应延时高 → 降低 limit，关闭 rerank

### 8.4 Item #4 Graph Memory

**决策：** MVP **暂不启用**；`memory_config.yaml` 预留 `graph_store` 配置点，用户可一键切换 `enabled: true` + 填 Neo4j 连接即可启用，**代码层零修改**（mem0 原生支持）。

**启用 vs 不启用的能力边界：**

| 场景 | 仅向量 | Graph Memory |
|---|---|---|
| 语义相似检索（"我妈怎么样"） | ✅ | ✅ |
| 关系聚合（"我养的动物有哪些"） | 🟡 可能漏召回 | ✅ 精确遍历 |
| 多跳推理（"家里谁有慢性病"） | ❌ 查询太抽象易漏 | ✅ 两跳查询命中 |
| 关系变更（分手/搬家/离职） | ❌ 旧记忆仍召回 | ✅ 可显式删边 |

**暂不启用的理由：**
1. MVP 阶段向量库已覆盖 80% 核心价值（用户事实、偏好、情感状态）
2. 多一次 LLM 调用会让 L3 触发时阻塞 2-5s
3. 多一个 Neo4j Docker 服务，初期部署复杂度不划算
4. 触发启用的反馈信号明确（用户抱怨"忘记家人"/"分手了还以为在恋爱"）→ 实测后再开

**未来开启的触发条件：**

- 用户反馈"AI 分不清我家里人的关系"
- 用户反馈"AI 记不住我宠物/朋友的具体信息"
- 需要跟踪长期关系演变（恋爱→分手、朋友→疏远）

**SaaS 模式下：** 此配置不生效；mem0 SaaS 是否内置 Graph 能力由 mem0 团队决定，用户无需配置。

### 8.5 Item #8 会话恢复完整性校验

**决策：** 轻量方案 — 以 `chat_history.json` 作为 source of truth，恢复时校验 `short_term_memory.json` 的一致性，不一致则从 chat_history 重建。不引入 WAL 日志或写入事务。

**设计原则：** "补齐数据 + 检查是否该触发压缩"，恢复逻辑与正常对话流程一致，不引入额外的恢复专用代码路径。

#### 三种崩溃场景及影响分析

| 场景 | 触发条件 | 影响 | 恢复策略 |
|------|----------|------|----------|
| **L3 写入中途崩溃** | L3 触发时做三件事：(a) LLM 压缩生成 block，(b) `mem0.add()` 写入长期记忆，(c) 更新 `short_term_memory.json`。任一步骤中途进程挂掉 | `short_term_memory.json` 可能截断损坏；`chat_history.json` 不受影响（独立 append）；mem0 可能写了也可能没写 | 从 chat_history 重建 short_term_memory；mem0 重复 add 无害（ADD-only 幂等） |
| **chat_history 与 short_term_memory 不一致** | 正常对话中（非 L3 触发时）崩溃，short_term_memory 的 `total_rounds` 落后于 chat_history 实际轮数 | short_term_memory 少了若干轮的 recent_messages | 从 chat_history 补回缺失轮次 |
| **mem0 写入失败** | L3 触发时 `mem0.add()` 失败（Qdrant 未启动、网络超时等），但 block 生成成功 | 这 20 轮的长期事实缺失于 mem0 向量库 | 会话关闭时的兜底写入（§4.3 `on_session_close`）会再尝试一次；mem0 ADD-only，重复写入无害 |

#### 恢复流程

```
恢复会话时：

1. 读取 short_term_memory.json
   ├─ json.load() 失败（文件损坏/截断）→ 进入【完全重建】
   └─ json.load() 成功 → 进入步骤 2

2. 校验一致性：short_term_memory.total_rounds vs chat_history 实际轮数
   ├─ 一致 → 正常恢复，无需额外操作
   └─ 不一致（chat_history 轮数 > total_rounds）→ 进入步骤 3

3. 增量修复：
   a. 从 chat_history 中取出"未被 short_term_memory 覆盖的轮次"
   b. 补入 recent_messages，更新 total_rounds
   c. 检查补回后是否达到 L3 触发条件（total_rounds % 26 == 0）
      ├─ 是 → 正常触发 L3（压缩 + mem0.add）
      └─ 否 → 不压缩，等后续对话自然触发

【完全重建】：
   a. 创建空的 short_term_memory 结构
   b. 从 chat_history 逐轮重放：L1 清洗 → 累积 → 达到 26 轮触发 L3 → 达到 4 个 block 触发 L4
   c. mem0.add() 对所有轮次（幂等，重复无害）
   d. 重建完成后正常恢复
```

#### 边界情况处理

| 边界情况 | 处理方式 |
|----------|----------|
| **L4 触发时崩溃** | 恢复时检查：active_blocks ≥ 4 且无对应 meta_block → 重新触发 L4。L4 输入是已持久化的 blocks，重跑安全 |
| **chat_history 本身损坏** | append-only JSON 数组，最多最后一条写到一半。尝试解析，失败则截断到最后一个完整的 `}`，丢失最多一条消息 |
| **首次启动（无历史文件）** | 不是恢复，是正常初始化：创建空 short_term_memory + 空 chat_history（含 metadata 行） |
| **完全重建的性能** | 需要对历史轮次逐批调用 LLM 做 L3 压缩。100 轮历史约需 4-5 次 L3 调用（每次压缩 20 轮），耗时约 10-20 秒。低频操作，可接受 |

#### 关键设计点

1. **chat_history 是 source of truth**：它是 append-only 的，天然比需要原地更新的 short_term_memory 更可靠
2. **恢复逻辑 = 正常逻辑**：不引入专门的恢复代码路径，而是复用"补齐数据 + 检查触发条件"的正常流程
3. **mem0 天然幂等**：v3 ADD-only 算法意味着重复写入只会多几条冗余事实，不会丢数据或覆盖
4. **不做预防性写入优化**：不引入 WAL、不做 fsync、不做写入事务。单用户场景下，崩溃是低概率事件，恢复成本（几秒到十几秒）远低于预防成本（代码复杂度）

### 8.6 Item #5 L3/L4 压缩 prompt 模板审视

**决策：** 对 §3.3 中的 L3/L4 prompt 草案做三项改进，设计层面闭环；具体参数（压缩比、摘要质量）待实现后实测微调。

**三项改进：**

1. **加入 `<analysis>` 草稿步骤**：让 LLM 先分析再输出摘要，减少遗漏。参考 Claude Code L4 prompt 的 `<analysis>` 设计（§9.5）。
2. **L3 明确保留优先级 P0-P3**：从原草案的平铺列表改为带优先级的结构，压缩时有明确的取舍依据。
3. **L4 增加明确的输出模板**：原草案只说"整合为更高层的记忆"，现在定义了与 L3 对应但层级不同的输出结构。

**改进后的 L3 prompt（取代 §3.3 中的原版）：**

```
你是一个对话记忆压缩器。请将以下 {N} 轮对话压缩为一段结构化摘要。

首先在 <analysis> 标签内梳理对话中的关键信息（此部分不会出现在最终摘要中）：
<analysis>
- 逐轮扫描，标记：情感变化点、用户事实、未完成事项、话题转折
- 识别哪些内容是重复/寒暄/可丢弃的
- 检查是否有被纠正的错误信息需要排除
</analysis>

然后基于分析，输出最终摘要。

保留优先级（从高到低）：
P0: 用户事实 & 偏好（"用户叫小明，养了一只橘猫叫团子"）
P1: 情感状态变化（"用户从开心变得沮丧，因为加班"）
P2: 未完成话题 & 承诺（"用户说明天要面试，让我提醒"）
P3: 话题转折点（"从聊天气转到聊工作压力"）

丢弃：寒暄、AI 冗长解释（只保留结论）、已被纠正的错误信息、重复讨论的同一话题。

输出格式：
## 对话摘要 (轮次 {start}-{end})
- 情感轨迹: ...
- 关键事实: ...
- 未完成话题: ...
- 用户偏好: ...
```

**改进后的 L4 prompt（取代 §3.3 中的原版）：**

```
你是一个长期记忆整合器。以下是 4 段对话摘要，覆盖 {total_rounds} 轮对话。
请整合为一段更高层的模式级记忆。

首先在 <analysis> 标签内分析跨摘要的模式（此部分不会出现在最终输出中）：
<analysis>
- 哪些事实在多个摘要中反复出现？→ 提炼为稳定特征
- 情感变化是否有趋势？→ 从单次情绪提炼为情绪模式
- 哪些话题反复被提起？→ 识别核心兴趣
- 用户与 AI 的互动方式有无变化？→ 判断关系阶段
- 哪些承诺跨多个摘要仍未完成？→ 标记为重要待办
</analysis>

然后基于分析，输出整合后的模式级摘要。

与事件级摘要的区别：不记录"发生了什么"，而是提炼"这意味着什么"。

输出格式：
## 长期模式摘要 (轮次 {start}-{end})
- 用户画像: ...（稳定的性格特征、生活状态）
- 情感趋势: ...（跨时段的情绪变化模式，而非单次情绪）
- 核心话题: ...（反复出现的兴趣和关注点）
- 关系阶段: ...（用户与 AI 的互动模式变化）
- 重要承诺: ...（跨多轮仍未完成的事项）
```

**输出格式决策：保持 markdown。** 理由：L3/L4 的输出有两个消费者 — 主聊天模型（每轮注入 context）和压缩模型（L4 合并 L3 blocks）。两者都是 LLM，markdown 可读性好、直接注入无需转换。JSON 格式反而会增加一层不必要的"JSON → 自然语言"转换。

**`<analysis>` 草稿块的处理：** 压缩模型输出中的 `<analysis>...</analysis>` 部分在存入 block 的 `summary` 字段前应被剥离（正则匹配删除），只保留最终摘要。这样不会增加 block 的 token 体积。

**待实测微调项（实现后验证）：**
- `<analysis>` 步骤对轻量模型（haiku 级别）的效果是否稳定
- 压缩比是否仍在 6:1 左右（L3）和 3:1 左右（L4）
- 输出模板的各字段是否被稳定填充（不出现空字段或格式偏移）

### 8.7 Item #6 L1 填充词表完善

**决策：** 调整初始词表（删除"然后"，新增"呃""额"），确立"宁可漏删不可误删"的设计原则。具体词表待实测 ASR 输出后根据实际情况扩充。

**词表调整：**

| 操作 | 词 | 理由 |
|------|-----|------|
| 删除 | "然后" | 在叙事语境中是有意义的连接词（"我先去了超市，然后去了银行"），不应无脑删除 |
| 新增 | "呃" | ASR 最常见的犹豫音之一 |
| 新增 | "额" | ASR 最常见的犹豫音之一 |
| 保留 | "嗯"、"啊"、"那个"、"就是说"、"对对对" | 在任何语境下都无语义价值的纯口头禅 |

**调整后的词表（已同步更新 §3.3 和 §7 中的配置示例）：**

```yaml
filler_words: ["嗯", "啊", "呃", "额", "那个", "就是说", "对对对"]
```

**设计原则：宁可漏删，不可误删。**

L1 的定位是轻量级纯规则清洗，不动用 LLM。填充词表只收录**在任何语境下都无语义价值**的纯口头禅。有歧义的词（如"然后"、"所以"、"反正"、"就是"）不入表，交给 LLM 在理解时自行忽略。

**理由：**
- L1 是不可逆操作（清洗后的文本写入 chat_history 的 `content` 字段，原始文本仅在 ASR 输入时保留于 `raw_input`）
- 误删有语义的词会导致 LLM 理解偏差，且无法恢复
- 漏删填充词的代价很低 — LLM 天然能忽略口头禅，最多浪费几个 token

**待实测扩充（实现后根据 ASR 输出调整）：**
- 观察实际 ASR 转写结果中的高频无意义词
- 根据使用的 ASR 引擎（sherpa_onnx / whisper.cpp）的输出特征调整
- 考虑是否需要按语言分表（中文 / 英文 / 日文）

### 9.7 6 种恢复策略（Recovery Strategies）

全部位于 `query.ts`，通过修改 `state.transition.reason` 标记恢复类型后 `continue` 回到循环顶部：

| 策略 | 位置 | 触发条件 | 恢复方式 |
|------|------|----------|----------|
| `collapse_drain_retry` | `query.ts:1087-1117` | `prompt_too_long` 且未曾排空 | 调用 `contextCollapse.recoverFromOverflow` 排空暂存的折叠 |
| `reactive_compact_retry` | `query.ts:1119-1162` | 仍然 `prompt_too_long` 或媒体超限 | 调用 `reactiveCompact.tryReactiveCompact` 让 Claude 生成完整摘要 |
| `max_output_tokens_escalate` | `query.ts:1210-1217` | 触及默认 8k 输出限制 | 升级到 64k 限制重试 |
| `max_output_tokens_recovery` | `query.ts:1238-1246` | 触及任何输出限制 | 注入"继续"提示重试（最多 3 次） |
| `stop_hook_blocking` | `query.ts:1290-1302` | Stop 钩子阻塞 | 将阻塞错误注入上下文重试 |
| `token_budget_continuation` | `query.ts:1331-1338` | 任务预算尚有剩余 | 注入预算提示继续执行 |

**对 emotion-robot 的借鉴：**
- 本项目初期 MVP **不需要**实现全部 6 种恢复，因为轮次驱动是 proactive 的（触发 L3 前不会遇到 prompt_too_long）
- 但应保留 1 种兜底：**单次 LLM 调用失败时的重试机制**（对应 `max_output_tokens_recovery` 的思路 —— 清理孤立消息 + 重试）
- 如果未来加入 token 兜底（待决事项 #3 的混合驱动），可补充 `reactive_compact_retry` 的等价物

### 9.8 实现顺序建议

基于 Claude Code 的依赖关系，emotion-robot 的 3 层压缩建议按以下顺序实现：

```
Step 1: L1 Snip 纯规则实现
        参考 services/compact/snipCompact.ts
        接口：(messages) => { messages, tokensFreed }
        
Step 2: L3 Collapse 的单次压缩实现（不做渐进/暂存）
        参考 services/compact/compact.ts:387 compactConversation
        参考 services/compact/prompt.ts 的 prompt 模板设计（但改为情感场景版）
        接口：(messages_20rounds) => block_summary

Step 3: short_term_memory.json 的读写逻辑
        参考 services/SessionMemory/sessionMemory.ts（如果相关）

Step 4: L4 Super-Compact
        参考 Step 2，但 prompt 改为"模式级"整合模板
        接口：(blocks_4) => meta_block

Step 5: 触发判断与调度
        在 Memory Manager 内实现，不需要 Claude Code 那样复杂的 state machine
        简单的 if rounds % 26 == 0 / if len(active_blocks) >= 4 即可
```
