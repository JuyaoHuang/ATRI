## 1. 模块概述

### 1.1 前端定位和职责

atri-webui 是 atri 项目的 Web 前端，采用**中心化架构**：后端管理所有 Provider 配置和 API Keys，前端只负责用户交互和数据展示。

**核心职责**：
- 提供聊天界面，支持实时流式对话（WebSocket）
- 提供角色选择和聊天历史管理
- 提供 11 个设置页面，管理各模块配置
- 集成 Live2D 模型渲染和表情控制
- 提供背景自定义、主题切换等个性化功能

**架构原则**：
- **后端管理配置**：LLM/TTS/ASR Provider 配置、API Keys 存储在后端 YAML，前端只读选择
- **前端管理偏好**：UI 偏好、背景设置、Live2D 开关存储在前端 LocalStorage
- **完全复用 AIRI UI**：UI 组件和 CSS 样式直接复用 AIRI，不自行调整样式

### 1.2 核心功能列表

1. **聊天界面**：流式消息显示、角色选择、历史对话切换
2. **双模式主页面**：Live2D 关闭时 ChatGPT 风格，开启时 AIRI 风格
3. **角色管理**：角色卡 CRUD（创建/编辑/删除/导入/导出）
4. **设置系统**：11 个设置页面，覆盖所有模块配置
5. **背景自定义**：图片上传、透明度/模糊度调节
6. **Live2D 集成**：模型渲染、表情控制、模型管理
7. **认证系统**：GitHub OAuth + 白名单模式
8. **数据管理**：导出/导入/清空聊天历史和记忆数据
9. **响应式设计**：支持桌面端、平板、手机

### 1.3 设计目标

- **零 CSS 调整**：UI 组件和样式完全复用 AIRI，不花时间调整 CSS
- **实时通信**：WebSocket 流式对话，指数退避重连
- **响应式**：支持移动端（sm/md/lg 断点）
- **易集成**：RESTful API + WebSocket，与后端解耦
- **可扩展**：组件化设计，易于添加新功能

### 1.4 与 AIRI 的核心差异

atri 前端与 AIRI 前端在功能上基本一致，但有以下关键差异：

#### 差异 1：双模式主页面

| 模式 | 触发条件 | 布局风格 | 样式来源 |
|------|---------|---------|---------|
| **模式 A：ChatGPT 风格** | Live2D 关闭 | 侧边栏 + 聊天区域 | agent 使用 `/frontend-design` skill 仿照 AIRI 风格实现 |
| **模式 B：AIRI 风格** | Live2D 开启 | Live2D 模型 + 浮动聊天框 | 直接复用 AIRI 实现 |

**模式 A（Live2D 关闭）**：
```
┌─────────────────────────────────────┐
│  Sidebar  │  Chat Area              │
│           │                         │
│  角色选择  │  消息列表                │
│  聊天历史  │                         │
│           │  输入框                  │
└─────────────────────────────────────┘
```
- 类似 ChatGPT 的标题/历史对话风格
- 已在 `atri-webui/src/` 中实现示例
- 样式需要 agent 使用 `/frontend-design` skill 仿照 AIRI 风格实现

**模式 B（Live2D 开启）**：
```
┌─────────────────────────────────────┐
│  [折叠面板]     Live2D 模型          │
│                                     │
│                                     │
│              ┌──────────────┐       │
│              │  聊天框（浮动）│       │
│              │  输入框       │       │
│              └──────────────┘       │
└─────────────────────────────────────┘
```
- 完全复用 AIRI 的主页面布局
- 左上角新增折叠面板组件

#### 差异 2：折叠面板组件（模式 B 专属）

Live2D 开启时，AIRI 没有"切换历史对话/切换角色"的功能。atri 需要新增一个**折叠面板组件**：

**功能需求**：
- 同时显示"历史对话列表"和"角色列表"
- 采用**单个折叠面板 + 标签页**设计（方案一）
- 位置：左上角
- 与 AIRI 风格契合

**交互设计**：
```
┌──────────────────┐
│ [折叠按钮 ▼]      │
├──────────────────┤
│ [对话] | [角色]   │  ← 标签页切换
├──────────────────┤
│ ● 今天的对话      │
│ ● 昨天的对话      │  ← 历史对话列表
│ ● ...            │
└──────────────────┘
```

**实现方式**：需要 agent 自行设计并实现，风格需与 AIRI 一致。

#### 差异 3：中心化架构

| 对比项 | AIRI（去中心化） | atri（中心化） |
|--------|----------------|---------------|
| API Key 存储 | 前端 LocalStorage | 后端 YAML 配置 |
| Provider 管理 | 前端完全控制 | 后端配置，前端只读选择 |
| 认证系统 | 无 | GitHub OAuth + 白名单 |
| 数据存储 | 前端 LocalStorage | 后端 API + 数据库 |

#### 差异 4：背景设置路由

AIRI 的 `/settings/scene` 路由待开发，atri 使用此路由实现聊天背景设置：
- 背景图片本地上传（FileReader API → base64 → LocalStorage）
- 透明度调节（0-100 滑块）
- 模糊度调节（0-10 滑块）
- 预设背景（纯色/渐变）

## 2. 技术选型

### 2.1 核心技术栈

| 类别 | 技术选择 | 版本 | 说明 |
|------|---------|------|------|
| **框架** | Vue 3 | ^3.5.32 | 组合式 API（Composition API） |
| **语言** | TypeScript | ~6.0.2 | 类型安全 |
| **构建工具** | Vite | ^8.0.9 | 快速开发、HMR |
| **状态管理** | Pinia | ^2.3.1 | Vue 3 官方推荐 |
| **样式** | UnoCSS | ^66.6.8 | 原子化 CSS |
| **组件库** | reka-ui | - | Headless 组件库 |
| **HTTP 客户端** | axios | ^1.15.1 | API 调用 |
| **路由** | vue-router | ^4.6.4 | SPA 路由 |
| **WebSocket** | 原生 WebSocket API | - | 实时通信 |
| **本地存储** | localforage | ^1.10.0 | IndexedDB 封装 |
| **工具库** | @vueuse/core | ^10.11.1 | Vue 组合式工具集 |
| **Live2D** | pixi-live2d-display | - | Live2D 模型渲染 |

### 2.2 技术选型依据

#### 2.2.1 Vue 3 + TypeScript

- **选择理由**：与 AIRI 技术栈一致，便于复用组件
- **组合式 API**：更好的逻辑复用（composables）
- **TypeScript**：类型安全，减少运行时错误

#### 2.2.2 Pinia

- **选择理由**：Vue 3 官方推荐的状态管理方案
- **优势**：TypeScript 原生支持、DevTools 集成、模块化设计
- **Store 划分**：chat、chats、characters、user、websocket、settings

#### 2.2.3 UnoCSS

- **选择理由**：与 AIRI 一致，原子化 CSS
- **优势**：按需生成、零运行时、响应式前缀（sm:/md:/lg:）
- **复用策略**：直接复制 AIRI 的 `uno.config.ts` 和主题配置

#### 2.2.4 reka-ui

- **选择理由**：Headless 组件库，提供无样式的交互组件
- **优势**：可访问性（a11y）、键盘导航、与 UnoCSS 配合
- **使用场景**：Dialog、Dropdown、Tabs、Switch 等交互组件

#### 2.2.5 localforage

- **选择理由**：IndexedDB 的简单封装，API 类似 localStorage
- **优势**：支持大数据存储、异步操作、自动降级
- **使用场景**：背景图片缓存、用户偏好、Live2D 开关状态

### 2.3 UI/CSS 复用策略

**核心原则：完全复用 AIRI 的 UI 实现，零 CSS 调整。**

#### 2.3.1 组件复用来源

| 来源 | 路径 | 复用内容 |
|------|------|---------|
| **AIRI 基础组件** | `airi/packages/ui/` | Button、Input、Modal、Dropdown 等 |
| **AIRI 业务组件** | `airi/packages/stage-ui/` | 聊天组件、设置页面组件 |
| **AIRI 主页面** | `airi/packages/stage-web/` | 主页面布局（模式 B） |
| **AIRI 设置页面** | `airi/packages/stage-pages/` | 设置页面布局和逻辑 |
| **AIRI Live2D** | `airi/packages/stage-ui-live2d/` | Live2D 渲染和表情控制 |

#### 2.3.2 样式复用

- **UnoCSS 配置**：直接复制 `airi/uno.config.ts`
- **主题变量**：复制颜色、字体、间距等主题配置
- **全局样式**：复制全局 CSS 文件

#### 2.3.3 需要 agent 实现的样式

以下部分无法直接复用 AIRI，需要 agent 使用 `/frontend-design` skill 仿照 AIRI 风格实现：

1. **模式 A（ChatGPT 风格）的聊天界面**：侧边栏 + 聊天区域布局
2. **折叠面板组件**：Live2D 模式下的历史对话/角色切换面板
3. **背景设置页面**：图片上传 + 透明度/模糊度滑块

### 3.3 数据流设计

#### 3.3.1 聊天数据流（单向）

```
用户输入文本
    ↓
InputBox.vue → emit('send', text)
    ↓
ChatArea.vue → useChatStore().sendMessage(text)
    ↓
chatStore → 添加用户消息到 messages[]
    ↓
wsStore → WebSocket 发送 { type: "input:text", data: { text, chat_id, character_id } }
    ↓
后端 ChatAgent 处理
    ↓
WebSocket 接收 { type: "output:chat:chunk", data: { chunk, chat_id } }
    ↓
chatStore → 追加 chunk 到 streamingText
    ↓
WebSocket 接收 { type: "output:chat:complete", data: { full_reply, chat_id } }
    ↓
chatStore → 将 streamingText 转为完整消息，添加到 messages[]
    ↓
MessageList.vue → 响应式更新显示
```

#### 3.3.2 数据存储策略

| 数据类型 | 存储位置 | 管理方式 | 说明 |
|---------|---------|---------|------|
| 聊天历史 | 后端 API | 只读展示 | `GET /api/chats` |
| 角色卡 | 后端 API | 完整 CRUD | `GET/POST /api/characters` |
| LLM Provider | 后端 YAML | 只读列表 | 前端不可修改 |
| TTS/ASR Provider | 后端 YAML | 只读选择 | 前端可切换，不可新增 |
| 背景设置 | 前端 LocalStorage | 完全控制 | base64 图片 + 透明度 |
| UI 偏好 | 前端 LocalStorage | 完全控制 | 语言、主题、字体 |
| Live2D 开关 | 前端 LocalStorage | 完全控制 | boolean |
| 用户认证 | 前端 Cookie/Storage | JWT Token | 后端签发 |

```typescript
// 发送消息（伪代码）
const sendMessage = async (text: string) => {
  // 1. 添加用户消息到本地
  addUserMessage(text)
  
  // 2. 开始流式输入
  startStreaming()
  
  // 3. 通过WebSocket发送
  await wsStore.sendMessage({
    type: 'input:text',
    data: { text, chat_id, character_id }

// 处理WebSocket消息（伪代码）
const handleWebSocketMessage = (message: any) => {
  switch (message.type) {
    case 'output:chat:chunk':
      appendStreamingText(message.data.chunk)
      break
    case 'output:chat:complete':
      finishStreaming()
      break
    case 'error':
      finishStreaming()
      addErrorMessage(message.data.message)
      break
  }
}
```

### 6.3 重连策略

#### 6.3.1 指数退避算法

**参数配置**：
```typescript
const reconnectConfig = {
  maxAttempts: 5,           // 最大重连次数
  initialDelay: 1000,       // 初始延迟（1秒）
  maxDelay: 30000,          // 最大延迟（30秒）
  backoffFactor: 2          // 退避因子
}
```

**延迟计算**：
```typescript
// 伪代码
delay = min(initialDelay * pow(backoffFactor, attempts - 1), maxDelay)

// 实际延迟序列：
// 第1次：1秒
// 第2次：2秒
// 第3次：4秒
// 第4次：8秒
// 第5次：16秒
// 第6次+：30秒（最大值）
```

#### 6.3.2 重连流程图

```
连接断开
  ↓
检查重连次数 < maxAttempts?
  ↓ 是
计算延迟时间
  ↓
等待延迟
  ↓
尝试重连
  ↓
成功? → 重置重连计数
  ↓ 否
重连次数+1 → 返回检查
  ↓ 否（达到最大次数）
停止重连，显示错误
```

**关键注意事项**：
- ⚠️ 重连成功后重置计数器
- ⚠️ 达到最大次数后停止重连，提示用户手动刷新
- ⚠️ 页面可见性变化时暂停/恢复重连

### 6.4 心跳机制

#### 6.4.1 心跳参数

```typescript
const heartbeatConfig = {
```

#### 6.4.2 心跳逻辑

```typescript
// 伪代码
startHeartbeat() {
  heartbeatInterval = setInterval(() => {
    if (connected) {
      send({ type: 'ping' })
      
      // 设置超时检测
      heartbeatTimeout = setTimeout(() => {
        missedHeartbeats++
        if (missedHeartbeats >= missedLimit) {
          // 心跳超时，主动断开重连
          disconnect()
          reconnect()
        }
      }, timeout)
    }
  }, interval)
}

// 收到pong时重置
onPong() {
  clearTimeout(heartbeatTimeout)
  missedHeartbeats = 0
}
```

**关键注意事项**：
- ⚠️ 连接成功后立即启动心跳
- ⚠️ 连接断开时停止心跳
- ⚠️ 收到 pong 后重置超时计数

### 6.5 消息队列

#### 6.5.1 队列设计

**目的**：缓存连接断开时的消息，连接恢复后自动发送

**实现要点**：
```typescript
// 伪代码
const messageQueue: ClientMessage[] = []

sendMessage(message: ClientMessage) {
  if (!connected) {
    // 连接断开，加入队列
    messageQueue.push(message)
    console.warn('WebSocket未连接，消息已加入队列')
    return
  }
  
  // 连接正常，直接发送
  ws.send(JSON.stringify(message))
}

// 连接恢复后发送队列消息
onConnected() {
  while (messageQueue.length > 0) {
    const message = messageQueue.shift()
    ws.send(JSON.stringify(message))
  }
}
```

**关键注意事项**：
- ⚠️ 队列大小限制（避免内存溢出）
- ⚠️ 队列持久化到 localStorage（可选）
- ⚠️ 发送失败的消息重新加入队列

### 6.6 错误处理

#### 6.6.1 错误类型

| 错误类型 | 触发条件 | 处理方式 |
|---------|---------|---------|
| **连接失败** | 无法建立连接 | 自动重连 |
| **连接超时** | 连接建立超时 | 自动重连 |
| **心跳超时** | 连续丢失心跳 | 主动断开重连 |
| **消息发送失败** | send()抛出异常 | 加入队列，重连后发送 |
| **消息解析失败** | JSON.parse()失败 | 记录日志，忽略消息 |
| **业务错误** | 服务端返回 error | 显示错误提示 |

#### 6.6.2 错误处理流程

```typescript
// 伪代码
ws.onerror = (error) => {
  console.error('WebSocket错误:', error)
  // 不主动断开，等待onclose触发重连
}

ws.onclose = (event) => {
  connected = false
  stopHeartbeat()
  
  // 判断关闭原因
  if (event.code === 1000) {
    // 正常关闭，不重连
    return
  }
  
  // 异常关闭，尝试重连
  handleReconnect()
}

ws.onmessage = (event) => {
  try {
    const message = JSON.parse(event.data)
    handleMessage(message)
  } catch (error) {
    console.error('消息解析失败:', error)
    // 忽略无效消息
  }
}
```

**关键注意事项**：
- ⚠️ 区分正常关闭和异常关闭
- ⚠️ 业务错误显示用户友好的提示
- ⚠️ 记录错误日志便于调试

## 10. 认证与授权

### 10.1 认证流程

**GitHub OAuth 流程**：
```
1. 用户点击"使用GitHub登录"
2. 跳转到GitHub授权页面
3. 用户授权后，GitHub重定向到回调URL（带code参数）
4. 前端将code发送到后端
5. 后端用code换取access_token
6. 后端返回JWT token给前端
7. 前端存储token到LocalStorage
```

**实现要点**：
```typescript
// stores/user.ts
export const useUserStore = defineStore('user', () => {
  const token = ref<string | null>(localStorage.getItem('auth_token'))
  const user = ref<User | null>(null)
  
  const loginWithGitHub = () => {
    const clientId = import.meta.env.VITE_GITHUB_CLIENT_ID
    const redirectUri = `${window.location.origin}/auth/callback`
    const authUrl = `https://github.com/login/oauth/authorize?client_id=${clientId}&redirect_uri=${redirectUri}`
    window.location.href = authUrl
  }
  
  const handleCallback = async (code: string) => {
    const { token: newToken, user: userData } = await authAPI.githubCallback(code)
    token.value = newToken
    user.value = userData
    localStorage.setItem('auth_token', newToken)
  }
  
  const logout = () => {
    token.value = null
    user.value = null
    localStorage.removeItem('auth_token')
  }
  
  return { token, user, loginWithGitHub, handleCallback, logout }
})
```

### 11.2 LocalStorage 管理

```typescript
// utils/storage.ts
export const storage = {
  get<T>(key: string, defaultValue?: T): T | null {
    try {
      const value = localStorage.getItem(key)
      return value ? JSON.parse(value) : defaultValue ?? null
    } catch {
      return defaultValue ?? null
    }
  },
  
  set(key: string, value: any): void {
    try {
      localStorage.setItem(key, JSON.stringify(value))
    } catch (error) {
      console.error('LocalStorage写入失败:', error)
    }
  },
  
  remove(key: string): void {
    localStorage.removeItem(key)
  },
  
  clear(): void {
    localStorage.clear()
  }
}
```

```typescript
class Logger {
```
