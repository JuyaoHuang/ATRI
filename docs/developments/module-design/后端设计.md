## 1. 技术选型

### 1.1 后端框架：FastAPI 

**决策：** 使用 FastAPI + Uvicorn 作为后端框架。

**参考项目对比：**

| 项目 | 语言 | 后端框架 | 通信方式 |
|------|------|----------|----------|
| OLV | Python | FastAPI + Uvicorn | WebSocket + REST |
| Neuro | Python | 无框架（纯 Python） | Socket.IO |
| airi | TypeScript | Hono + crossws | WebSocket + HTTP |

**选择 FastAPI 的理由：**
- 原生 async — LLM 流式输出、mem0 调用都是异步的
- WebSocket 支持 — 后续加语音流（ASR/TTS）需要
- 自动生成 API 文档（Swagger UI）— 调试方便
- OLV 已验证此方案可行
- Python 生态 — 与 mem0、openai SDK、各种 ASR/TTS 库天然兼容

### 1.2 Python 版本 + 依赖管理：Python 3.11+ / uv 

**决策：** Python 3.11+，使用 uv 进行依赖管理和版本管理。

**Python 3.11+ 的理由：**
- 性能比 3.10 提升 10-60%（CPython 优化）
- `tomllib` 内置（读取 pyproject.toml 不需要额外依赖）
- `asyncio.TaskGroup` 可用（并发任务管理更优雅）
- mem0 支持 3.9~3.12，FastAPI 支持 3.8+，均兼容

**uv 的理由：**
- OLV 已验证（`uv sync` 一键安装）
- Rust 实现，比 pip 快 10-100x
- 兼容 pip 生态，`pyproject.toml` + `uv.lock`
- 同时管理 Python 版本（`uv python install 3.11`）

**项目初始化命令：**
```bash
uv init emotion-robot
uv python pin 3.11
uv add fastapi uvicorn loguru pyyaml openai mem0ai
```

**部署方案：** 开发阶段用 uv 管理，项目完成后追加 docker-compose 一键部署。Docker 配置不在 MVP 范围内。

### 1.3 前端：独立仓库 

**决策：** 前端作为独立仓库，不包含在后端项目中。

**理由：**
- 后端 Python（FastAPI），前端可用任何技术栈（计划仿照 airi 的 Vue 实现）
- 独立部署、独立版本管理
- 通过 WebSocket + REST API 通信，接口清晰
- OLV 也是前端独立（Git 子模块），验证过此模式可行

### 2.3 模块间依赖关系

```
service_context.py（顶层编排）
  ├── llm/factory.py        → 创建各出口的 LLM 实例
  ├── memory/manager.py     → 注入 L3/L4 用的 LLM 实例 + mem0 配置
  ├── agent/chat_agent.py   → 注入主聊天 LLM 实例 + Memory Manager
  ├── asr/factory.py        → 创建 ASR 实例（预留）
  └── tts/factory.py        → 创建 TTS 实例（预留）

agent/chat_agent.py
  ├── 依赖 llm/interface.py（主聊天 LLM）
  └── 依赖 memory/manager.py（记忆管理）

memory/manager.py
  ├── 依赖 llm/interface.py（L3/L4 压缩用的 LLM）
  ├── 依赖 memory/short_term.py
  ├── 依赖 memory/long_term.py（mem0）
  ├── 依赖 memory/compressor.py
  └── 依赖 memory/snip.py（L1）
```

### 2.4 设计原则

- **每个模块遵循相同的三层结构**：`interface.py`（抽象基类）→ `providers/`（具体实现）→ `factory.py`（注册表工厂）
- **模块间通过接口依赖**：`chat_agent` 依赖 `LLMInterface`，不依赖具体的 `OpenAICompatibleLLM`
- **配置与代码分离**：所有可配置项在 `config/` 下的 YAML 文件中，代码中不硬编码
- **预留模块只建骨架**：ASR/TTS/translate 只创建 `interface.py` 和 `factory.py`，不写实现代码

### 2.5 核心数据流（ChatAgent + LLM calling + Memory system + raw input）

> **补充注记（2026-04-19）**：本节明确 Phase 4 引入 `ChatAgent` 后四方的协作关系，
> 详细逐步图见 `docs/记忆系统设计讨论.md` §6.1。

**主响应路径**：

```
raw user text (文字 / ASR)
     │
     ▼
┌──────────────── ChatAgent (src/agent/chat_agent.py) ──────────────────┐
│  组合层：持有 persona + LLMInterface(role=chat) + MemoryManager        │
│                                                                        │
│  (1) messages = await mgr.build_llm_context(raw, persona)              │
│         └→ MemoryManager 内部组装 6 段 payload（§3.5 顺序）：           │
│            [1] system=persona                                          │
│            [2] 长期事实 = search_long_term(raw) → mem0.search(...)     │
│            [3] meta_blocks (cleaned)                                   │
│            [4] active_blocks (cleaned)                                 │
│            [5] recent_messages (cleaned)                               │
│            [6] 本轮 user_input (raw, 原样)                             │
│                                                                        │
│  (2) async for chunk in llm.chat_completion_stream(messages):          │
│         yield chunk  (LLM 原生语言文本，如 zh)                          │
│            ├→ 前端 plugin       (默认消费 LLM 原文)                     │
│            ├→ 翻译模块 plugin   (可选启用：output text →                │
│            │                     translate → translated text；         │
│            │                     下游消费者 = 前端 / TTS / 两者)         │
│            └→ TTS 模块 plugin   (可选启用；消费源 = LLM 原文 /          │
│                                  翻译模块输出，由配置决定)              │
│                                                                        │
│  (3) stream 结束后自动调: mgr.on_round_complete(raw_user, reply)       │
│         └→ MemoryManager 内部:                                         │
│            ├ L1 snip(user)          → cleaned                          │
│            ├ chat_history.append    (cleaned + 可选 raw_input)         │
│            ├ recent_messages +=     (cleaned, reply)                   │
│            ├ total_rounds % 26 == 0 → L3 Collapse + mem0.add           │
│            └ active_blocks ≥ 4      → L4 Super-Compact                 │
└────────────────────────────────────────────────────────────────────────┘
```

**四方职责边界**：

| 模块 | 依赖 | 被谁调用 | 做什么 |
|------|------|---------|--------|
| **raw input text** | 上游（Phase 5 FastAPI WS / CLI / 其他入口） | — | 原始文本字符串，进 `ChatAgent.chat()` |
| **ChatAgent** (`src/agent/`) | `LLMInterface` + `MemoryManager` + `persona` | 上游入口 | 组合层——组装 payload → 调 LLM → 分发 stream → 提交回合 |
| **LLM calling** (`src/llm/`) | 无（stateless 接口） | ChatAgent（主聊天 role=chat） + MemoryManager（L3/L4 压缩 role=l3_compress / l4_compact） | 流式 + 非流式文本生成；接口层只抛异常 |
| **Memory system** (`src/memory/`) | `LLMInterface`（压缩用）+ `mem0` | ChatAgent（build_llm_context / on_round_complete / search_long_term / append_system_note） | L1/L3/L4 压缩 + chat_history 归档 + 短/长期记忆管理 |

**可插拔 plugin 模块（LLM stream 旁路消费者）**：

| Plugin | 启停 | 消费源 | 位置 | 影响记忆系统？ |
|--------|------|--------|------|---------------|
| 前端 | 上游决定 | LLM 原文（默认）/ 翻译输出（可配） | 上游 | 否 |
| 翻译模块 | 配置开关 | LLM 原文 → 翻译后文本 | `src/translate/`（预留） | 否（不入 chat_history） |
| TTS | 配置开关 | LLM 原文 / 翻译输出（可配） | `src/tts/`（预留） | 否 |

**关键约束**：
- **L1 位置**：`MemoryManager.on_round_complete` 的**内部步骤**，不是独立入口层。ChatAgent 透传 raw user_input，MemoryManager 内部 L1。这让 LLM 位置[6] 保留 raw（当轮鲜活），chat_history / recent_messages / mem0 索引全部为 cleaned（跨轮一致）。
- **翻译模块 ≠ TTS 前置**：翻译模块是作用于 LLM 输出文本的独立转换器，下游消费者（前端 / TTS / 两者）由配置决定，翻译模块本身不知道谁消费。
- **chat_history 语言不可变**：无论前端 / 翻译 / TTS 怎么配，chat_history 始终记录 LLM 原生语言，确保记忆 / 压缩 / 恢复的语言一致性。
- **错误路径**：LLM 抛 `LLMError` → ChatAgent yield 错误文本给调用方 + 调 `mgr.append_system_note(...)` 写入 chat_history 的 `role=system` 行，不计轮、不触发 L3、不入 recent_messages。

## 3. 日志方案

### 3.1 技术选型：loguru 

**决策：** 沿用 OLV 的日志方案 — loguru + 控制台/文件双输出。

**为什么用 loguru 而不是标准 logging：**
- 零配置：`from loguru import logger` 即可使用，不需要 `getLogger` / `Handler` / `Formatter` 的繁琐设置
- 自动带颜色、自动格式化异常堆栈
- `logger.catch` 装饰器可以自动捕获函数异常并记录
- OLV 已验证此方案在同类项目中可行

### 3.2 日志配置

```python
# src/utils/logger.py
import sys
from loguru import logger

def init_logger(console_log_level: str = "INFO") -> None:
    logger.remove()  # 移除默认 handler

    # 控制台输出（带颜色）
    logger.add(
        sys.stderr,
        level=console_log_level,
        format="<green>{time:YYYY-MM-DD HH:mm:ss}</green> | "
               "<level>{level: <8}</level> | "
               "<cyan>{name}</cyan>:<cyan>{function}</cyan>:<cyan>{line}</cyan> | "
               "{message}",
        colorize=True,
    )

    # 文件输出（带毫秒 + extra 字段）
    logger.add(
        "logs/debug_{time:YYYY-MM-DD}.log",
        rotation="10 MB",
        retention="30 days",
        level="DEBUG",
        format="{time:YYYY-MM-DD HH:mm:ss.SSS} | {level: <8} | "
               "{name}:{function}:{line} | {message} | {extra}",
        backtrace=True,
        diagnose=True,
    )
```

### 3.4 使用方式

```python
# 每个模块文件直接导入使用
from loguru import logger

class MemoryManager:
    def on_round_complete(self, round_num: int):
        logger.info(f"Round {round_num} complete")
        if round_num % 26 == 0:
            logger.info(f"Triggering L3 compress for rounds {round_num-19}-{round_num}")
```

### 3.5 配置项

| 配置 | 值 | 说明 |
|------|-----|------|
| 控制台级别 | INFO（默认）/ DEBUG（`--verbose`） | 命令行参数切换 |
| 文件级别 | DEBUG | 始终记录详细日志 |
| 文件轮转 | 10 MB | 单文件超过 10MB 自动轮转 |
| 文件保留 | 30 天 | 超过 30 天的日志自动删除 |
| 文件路径 | `logs/debug_{date}.log` | 按日期分文件 |
