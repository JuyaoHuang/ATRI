```python
# src/tts/interface.py
class TTSInterface(ABC):
    @abstractmethod
    async def synthesize(self, text: str, voice_id: Optional[str] = None, **kwargs) -> bytes:
        """同步合成语音（返回完整音频数据）"""
        raise NotImplementedError

    @abstractmethod
    async def synthesize_stream(self, text: str, voice_id: Optional[str] = None, **kwargs) -> AsyncIterator[bytes]:
        """流式合成语音（返回音频数据流）"""
        raise NotImplementedError

    @abstractmethod
    async def get_voices(self) -> list[dict]:
        """获取可用声音列表"""
        raise NotImplementedError
```

#### 1. 架构决策：装饰器工厂模式 vs OLV 的 if-elif 工厂

**核心决策**：采用 atri 后端 LLMFactory 的装饰器注册模式，而非直接复用 OLV 的 if-elif 工厂。

**对比分析**（对话中的表格）：

| 维度         | OLV 工厂模式                     | atri 装饰器模式                        |
| ------------ | -------------------------------- | -------------------------------------- |
| **注册方式** | 静态 if-elif 分支                | `@Factory.register("name")` 装饰器     |
| **扩展性**   | 每增加 provider 需修改工厂类     | 新增 provider 文件即可，工厂类无需改动 |
| **代码位置** | 工厂类集中管理所有 provider 导入 | provider 文件自行注册，解耦            |

**决策理由**：

1. **一致性**：atri 后端已在 LLM 层验证装饰器模式（Phase 2），ASR/TTS 应保持架构一致
2. **可维护性**：OLV 的 `tts_factory.py` 有 228 行，19 个 elif 分支；装饰器模式下工厂类仅 60 行且永不修改
3. **插件化**：装饰器模式天然支持插件式扩展，符合 readme.md 中"插件式模块"的设计目标

**参考代码路径**：
