## 1. 项目概览

### 1.1 项目简介

**atri** 是一个基于 RAG（检索增强生成）的情感聊天机器人，专注于**记忆可持续化**和**多模态交互**。项目采用前后端分离架构，后端提供 REST API + WebSocket 服务，前端支持双模式 UI（ChatGPT 风格 / AIRI 风格）。

**核心特性**：
- **长期记忆系统**：基于 mem0 的 RAG 架构，支持会话恢复和跨会话记忆
- **多角色对话**：支持多个聊天角色（Persona），每个角色独立记忆
- **流式对话**：WebSocket 实时流式输出，支持打字机效果
- **模块化设计**：LLM/ASR/TTS/翻译模块均采用工厂模式，可插拔配置
- **Live2D 集成**：可选的 Live2D 模型展示（复用 AIRI 实现）

## 3. 架构分层

### 3.1 后端分层架构

后端采用**分层模块化架构**，每层职责清晰，模块间通过接口依赖。

```
┌─────────────────────────────────────────────────────────────┐
│                    Service Layer (FastAPI)                   │
│  REST API + WebSocket + 路由 + 生命周期管理                   │
│  详见: docs/后端API接口文档.md                                 │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      Agent Layer (ChatAgent)                 │
│  组合 LLM + Memory + Persona，处理主聊天逻辑                  │
│  详见: docs/后端设计.md §2.5                                  │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────┬──────────────────────────────────────┐
│   Core Layer (LLM)   │   Memory Layer (RAG)                 │
│   LLM 调用 + 流式    │   L1/L3/L4 压缩 + mem0 长期记忆      │
│   详见: LLM调用层    │   详见: 记忆系统设计讨论.md           │
└──────────────────────┴──────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    Storage Layer (存储抽象)                   │
│  ChatStorageInterface + JSONChatStorage                      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────┬──────────────────────────────────────┐
│  Input Layer (ASR)   │   Output Layer (TTS + 翻译)          │
│  语音输入（预留）     │   语音输出 + 文本翻译（预留）          │
│  详见: ASR模块设计   │   详见: TTS模块设计文档.md            │
└──────────────────────┴──────────────────────────────────────┘
```

#### 3.1.1 Core Layer - LLM 调用层

**职责**：封装 LLM 调用逻辑，提供统一的流式/非流式接口。

**核心组件**：
- `LLMInterface`：抽象基类，定义 `chat_completion_stream` 流式接口
- `LLMFactory`：注册表工厂，装饰器注册 Provider
- `openai_compatible`：MVP 实现，支持 OpenAI 兼容 API

**设计特点**：
- 注册表工厂模式：Provider 通过 `@LLMFactory.register("name")` 自注册
- 流式优先：子类只需实现流式接口，非流式有默认实现
- 凭证池 + 角色映射：`llm_configs` 池 + `llm_roles` 映射 3 个出口（chat/l3_compress/l4_compact）

**详细设计**：`D:\Coding\GitHub_Resuorse\emotion-robot\docs\LLM调用层设计讨论.md`

#### 3.1.2 Memory Layer - 记忆系统

**职责**：管理短期记忆（L1/L3/L4 压缩）和长期记忆（mem0 RAG）。

**核心组件**：
- `MemoryManager`：调度 L1/L3/L4 压缩 + 轮次计数 + 会话生命周期
- `ShortTermStore`：短期记忆读写（`short_term_memory.json`）
- `LongTermMemory`：mem0 集成封装（双模式：sdk / local_deploy）
- `Compressor`：L3/L4 压缩逻辑（调用 LLM 生成摘要）
- `Snip`：L1 纯规则清洗（填充词去除、连续重复去重）
- `ChatHistoryWriter`：chat_history.json 读写（append-only）

**设计特点**：
- **三层压缩**：L1 规则清洗 → L3 单次压缩（20 轮 → 1 block）→ L4 合并压缩（4 blocks → 1 meta_block）
- **触发调度**：`total_rounds % 26 == 0` 触发 L3，`len(active_blocks) >= 4` 触发 L4
- **mem0 集成**：L3 触发时同步调用 `mem0.add(raw_20_rounds)`，每轮调用前 `mem0.search(current_input)`
- **会话恢复**：以 `chat_history` 为 source of truth，恢复时校验 `short_term_memory` 一致性

**详细设计**：`D:\Coding\GitHub_Resuorse\emotion-robot\docs\记忆系统设计讨论.md`

#### 3.1.3 Agent Layer - 聊天代理

**职责**：组合 LLM + Memory + Persona，处理主聊天逻辑。

**核心组件**：
- `ChatAgent`：流式 `chat()` + 收集 `chat_collect()`，自动调用 `on_round_complete`
- `Persona`：Markdown + frontmatter 格式的角色定义
- `ServiceContext`：管理多 character 生命周期（懒加载缓存 + `close_all`）

**设计特点**：
- **双接口**：`chat()` 流式生成器 + `chat_collect()` 默认收集实现
- **自动记忆**：stream 结束后自动调用 `mgr.on_round_complete(raw_user, reply)`
- **错误处理**：`LLMError` → `append_system_note` 写 chat_history 的 `role=system` 行，不计轮、不触发 L3

**详细设计**：`D:\Coding\GitHub_Resuorse\emotion-robot\docs\后端设计.md` §2.5

#### 3.1.4 Service Layer - FastAPI 服务

**职责**：提供 REST API + WebSocket 端点，管理服务生命周期。

**核心组件**：
- `app.py`：FastAPI 应用工厂函数 + lifespan 管理
- `routes/`：健康检查、角色管理、聊天 CRUD、WebSocket 流式对话
- `main.py`：Uvicorn 服务器启动入口

**设计特点**：
- **REST API**：`GET /health`、`GET /api/characters`、`POST /api/chats`、`GET /api/chats/{id}`、`POST /api/chats/{id}/update`、`POST /api/chats/{id}/delete`
- **WebSocket**：`/ws` 流式对话端点（`input:text` → `output:chat:chunk` → `output:chat:complete`）
- **CORS 支持**：允许前端跨域访问
- **LLM 标题生成**：创建聊天时基于首条用户消息生成标题，失败回退截取前 20 字

**详细设计**：`D:\Coding\GitHub_Resuorse\emotion-robot\docs\后端API接口文档.md`

#### 3.1.5 Storage Layer - 存储抽象

**职责**：提供聊天存储的抽象接口，支持多种存储后端。

**核心组件**：
- `ChatStorageInterface`：ABC 定义 7 个方法（create/get/list/update/delete/append_message/get_messages）
- `JSONChatStorage`：文件持久化实现（`{base_path}/{user_id}/{character_id}/index.json` + `sessions/{chat_id}.json`）
- `DatabaseChatStorage`：数据库实现骨架（Phase 7 预留）

**设计特点**：
- **存储结构**：character-scoped 索引（每 character 一个 `index.json`）+ per-session 消息文件
- **原子写入**：使用 `atomic_replace` 保证写入原子性（Windows Defender 友好）

#### 3.1.6 Input/Output Layer - ASR/TTS（预留）

**ASR（语音输入）**：
- **5 个 Provider**：faster_whisper、whisper_cpp、sherpa_onnx_asr、azure_asr、Web Speech API
- **工厂模式**：装饰器注册（与 LLM 一致）
- **健康检查**：接口层定义 `async def health_check() -> bool`
- **详细设计**：`D:\Coding\GitHub_Resuorse\emotion-robot\docs\ASR模块设计文档.md`

**TTS（语音输出）**：
- **6 个 Provider**：edge_tts、cosyvoice2_tts、gpt_sovits_tts、siliconflow_tts、openai_tts、elevenlabs_tts
- **工厂模式**：装饰器注册（与 LLM 一致）
- **VAD 集成**：语音活动检测，优化静音处理
- **详细设计**：`D:\Coding\GitHub_Resuorse\emotion-robot\docs\TTS模块设计文档.md`

**翻译模块**：
- **Plugin 式设计**：独立 text transformer，不是 TTS 的前置层
- **消费者配置**：`output text → translate → translated text`，下游消费者（前端/TTS/两者）由配置决定

## 4. 核心数据流

### 4.1 后端核心数据流

**ChatAgent + LLM calling + Memory system + raw input**

```
raw user text (文字 / ASR)
     │
     ▼
┌──────────────── ChatAgent (src/agent/chat_agent.py) ──────────────────┐
│  组合层：持有 persona + LLMInterface(role=chat) + MemoryManager        │
│                                                                        │
│  (1) messages = await mgr.build_llm_context(raw, persona)              │
│         └→ MemoryManager 内部组装 6 段 payload（§3.5 顺序）：           │
│            [1] system=persona                                          │
│            [2] 长期事实 = search_long_term(raw) → mem0.search(...)     │
│            [3] meta_blocks (cleaned)                                   │
│            [4] active_blocks (cleaned)                                 │
│            [5] recent_messages (cleaned)                               │
│            [6] 本轮 user_input (raw, 原样)                             │
│                                                                        │
│  (2) async for chunk in llm.chat_completion_stream(messages):          │
│         yield chunk  (LLM 原生语言文本，如 zh)                          │
│            ├→ 前端 plugin       (默认消费 LLM 原文)                     │
│            ├→ 翻译模块 plugin   (可选启用：output text →                │
│            │                     translate → translated text；         │
│            │                     下游消费者 = 前端 / TTS / 两者)         │
│            └→ TTS 模块 plugin   (可选启用；消费源 = LLM 原文 /          │
│                                  翻译模块输出，由配置决定)              │
│                                                                        │
│  (3) stream 结束后自动调: mgr.on_round_complete(raw_user, reply)       │
│         └→ MemoryManager 内部:                                         │
│            ├ L1 snip(user)          → cleaned                          │
│            ├ chat_history.append    (cleaned + 可选 raw_input)         │
│            ├ recent_messages +=     (cleaned, reply)                   │
│            ├ total_rounds % 26 == 0 → L3 Collapse + mem0.add           │
│            └ active_blocks ≥ 4      → L4 Super-Compact                 │
└────────────────────────────────────────────────────────────────────────┘
```

**关键路径说明**：
1. **上下文构建**：`build_llm_context` 按严格顺序组装 6 段 payload（system → 长期事实 → meta → active → recent → 本轮输入）
2. **流式输出**：LLM 原生语言文本（如中文）通过 `yield chunk` 流式返回
3. **下游消费**：前端/翻译/TTS 均为独立 plugin 式模块，启停和消费源由配置决定
4. **自动记忆**：stream 结束后自动触发 `on_round_complete`，执行 L1 清洗 → 写入 chat_history → 检查 L3/L4 触发条件

### 4.2 记忆系统数据流

```
用户输入（文字 / ASR，raw text）
  │
  ▼
ChatAgent（Phase 4 组合层）
  │  持有: persona + LLMInterface(role=chat) + MemoryManager
  │
  ├── (1) await mgr.build_llm_context(raw user_input, persona)
  │        │
  │        ▼ 内部组装 payload（严格按 §3.5 顺序）：
  │        ┌────────────────────────────────────────────────────┐
  │        │ [1] system                : persona                │
  │        │ [2] 长期事实              : search_long_term(raw)  │
  │        │     └→ mem0.search(query=raw, user_id, agent_id)   │
  │        │        → 包装 "关于这位用户，你记得：\n- ..."       │
  │        │ [3] meta_blocks   (cleaned; L4 产物)                │
  │        │ [4] active_blocks (cleaned; L3 产物)                │
  │        │ [5] recent_messages (cleaned; §3.2 有效轮)          │
  │        │ [6] 本轮 user_input (raw, 原样透传)                 │
  │        └────────────────────────────────────────────────────┘
  │
  ├── (2) async for chunk in llm.chat_completion_stream(messages):
  │        yield chunk   (LLM 原生语言文本，如 zh)
  │            │
  │            ▼ 下游消费者（均为独立 plugin 式模块，启停 + 消费源由配置决定）：
  │            ├→ 前端（默认消费 LLM 原文；可配置改消费翻译后的文本）
  │            ├→ 翻译模块（可选启用；形态:output text -> translate -> translated，
  │            │             启用后其下游消费者 = 前端 / TTS / 两者，由配置决定）
  │            └→ TTS 模块（可选启用；消费源 = LLM 原文 / 翻译后文本，由配置决定）
  │
  ▼ stream 结束（全部 chunks 合并为 reply）
  │
  └── (3) await mgr.on_round_complete(
             user_msg={role:"human", content: raw, raw_input: ASR 原文?},
             ai_msg  ={role:"ai",    content: reply},
         )   ← ChatAgent 自动调用（S1b 决策）
          │
          ▼ MemoryManager.on_round_complete 内部：
          │
          ├── L1 Snip（仅作用于 user_msg；AI 回复不动）
          │   → cleaned_user
          │
          ├── chat_history.append_human(cleaned.content, raw_input=raw_input?)
          ├── chat_history.append_ai(reply)
          │     （错误轮次 S4 改走 append_system_note，不走 append_ai）
          │
          ├── recent_messages += [{human: cleaned}, {ai: reply}]
          │
          ├── total_rounds++（仅当 _is_valid_round：content 非空 &
          │                    不以 "Error" 开头；§3.2）
          │
          ├── 检查 total_rounds % 26 == 0?
          │     └─ 是 → L3 Collapse
          │              ├→ 最早 20 轮 cleaned → 压缩 LLM → block → active_blocks
          │              └→ 同一窗口 → long_term.add()
          │                 ↓ mem0 内部调 LLM 抽事实 → 事实句 embedding 入索引
          │
          └── 检查 active_blocks ≥ 4?
                └─ 是 → L4 Super-Compact
                         └→ 4 blocks → 压缩 LLM → meta_block → meta_blocks
```

**关键路径说明**：
1. **上下文构建**：6 段 payload 严格按顺序组装，长期事实通过 `mem0.search(raw)` 检索
2. **L1 清洗**：仅作用于用户消息（填充词去除、连续重复去重、超长截断），AI 回复不动
3. **轮次计数**：只有有效轮（content 非空 & 不以 "Error" 开头）才计入 `total_rounds`
4. **L3 触发**：每 26 轮触发一次，最早 20 轮 → 压缩 LLM → block → `active_blocks`，同时调用 `mem0.add()` 写入长期记忆
5. **L4 触发**：`active_blocks` 累积到 4 个时触发，4 blocks → 压缩 LLM → meta_block → `meta_blocks`
6. **会话恢复**：以 `chat_history` 为 source of truth，恢复时校验 `short_term_memory` 一致性，JSON 损坏则从 `chat_history` 完全重建
